# SVN Dev

기능 개발의 전체 라이프사이클을 하나의 워크플로우로 자동화합니다. 분석 문서 생성부터 브랜치 생성, 개발, 테스트, 머지까지 연결됩니다.

## 트리거

다음 문구가 포함되면 이 스킬을 자동 호출:
- "svn dev"

## 사용법

```
/svn-dev "로그인 모듈에 OAuth2 인증 추가"
/svn-dev "주문 목록 페이지 페이지네이션 구현"
/svn-dev "결제 모듈 타임아웃 버그 수정"
```

## 워크플로우

### Phase 1: 분석

```
1. 사용자 요구사항 파악
   ├─ 입력된 설명에서 기능 범위 추출
   └─ 필요 시 사용자에게 추가 질문 (모호한 부분 해소)

2. 코드베이스 분석
   ├─ 관련 파일/모듈 탐색
   ├─ 기존 코드 구조 파악
   └─ 영향 범위(변경 대상 파일) 식별

3. 구현 계획 수립
   ├─ 변경 사항을 단계별로 분리
   ├─ 각 단계의 작업 내용 + 대상 파일 정리
   └─ 테스트 계획 포함

4. 분석 문서 생성
   ├─ 옵시디언 문서로 저장 (obsidian-doc 스킬 연동)
   │   경로: 01. Projects/{YYYY-MM-DD}_{기능요약}.md
   └─ 사용자 확인: "분석 내용이 맞습니까? 개발을 진행할까요?"
```

### Phase 2: 브랜치 생성

```
1. 브랜치명 자동 생성
   ├─ 분석 내용에서 키워드 추출
   ├─ 형식: feature-{키워드} 또는 fix-{키워드}
   └─ 사용자 확인: "브랜치명 feature-oauth2-login 으로 생성할까요?"

2. svn-flow branch 호출
   ├─ 현재 브랜치를 원본으로 기록 (.svn-flow-origin)
   ├─ svn copy → 브랜치 생성
   └─ svn switch → 새 브랜치로 이동
```

### Phase 3: 개발

```
1. 분석 문서의 구현 계획을 순차 실행
   ├─ 단계별로 코드 변경
   ├─ 각 단계 완료 시 중간 커밋 (svn-commit 연동)
   └─ 주요 변경 전 사용자 확인

2. 사용자 개입 지점
   ├─ 파일 구조 변경 (신규 파일, 삭제) → 확인
   ├─ 외부 의존성 추가 → 확인
   └─ 기존 인터페이스 변경 → 확인
```

### Phase 4: 테스트

```
1. 테스트 실행
   ├─ 프로젝트의 테스트 명령 자동 감지 (make test, pytest 등)
   ├─ 테스트가 없으면 수동 확인 요청
   └─ 린트/타입체크도 함께 실행 (가능한 경우)

2. 실패 시 수정 루프
   ├─ 실패 원인 분석
   ├─ 코드 수정
   ├─ 재테스트
   └─ 최대 3회 반복 후 사용자에게 보고

3. 전체 통과 → Phase 5로 진행
```

### Phase 5: 완료 및 머지

```
1. 개발 결과 요약
   ├─ 변경 파일 목록 + 각 파일 변경 요약
   ├─ 커밋 이력
   └─ 테스트 결과

2. 사용자 확인
   └─ "머지를 진행하시겠습니까?"
       ├─ Yes → svn-flow merge 호출 (원본 브랜치 복귀 → 머지 → 브랜치 삭제)
       ├─ No → 브랜치 유지, 나중에 /svn-flow merge로 수동 진행
       └─ 추가 작업 → Phase 3로 복귀
```

## Phase 간 전환 규칙

| 전환 | 조건 | 사용자 확인 |
|------|------|------------|
| Phase 1 → 2 | 분석 문서 생성 완료 | "개발을 진행할까요?" |
| Phase 2 → 3 | 브랜치 생성 + 전환 완료 | 자동 진행 |
| Phase 3 → 4 | 구현 계획 전체 완료 | 자동 진행 |
| Phase 4 → 5 | 테스트 전체 통과 | "머지를 진행하시겠습니까?" |
| Phase 4 → 3 | 테스트 실패 (자동 수정 불가) | "수정 방향을 알려주세요" |
| Phase 5 → 3 | 추가 작업 요청 | 사용자 요청 시 |

## 스킬 연동

```
svn-dev
  ├── obsidian-doc (Phase 1 - 분석 문서 생성)
  ├── svn-flow branch (Phase 2 - 브랜치 생성)
  ├── svn-commit (Phase 3 - 중간 커밋)
  ├── svn-status (Phase 3/4 - 상태 확인)
  └── svn-flow merge (Phase 5 - 머지 + 브랜치 삭제)
```

## 옵션

| 인자 | 설명 |
|------|------|
| `"설명"` | 개발할 기능/버그 설명 (필수) |
| `--skip-doc` | 분석 문서 생성 생략 |
| `--skip-branch` | 브랜치 생성 생략 (현재 브랜치에서 바로 작업) |
| `--skip-test` | 테스트 Phase 생략 |
| `--no-merge` | 머지 없이 개발+테스트만 진행 |
| `--branch name` | 브랜치명 직접 지정 |

## 중단 및 재개

- 작업 중 중단: 현재까지 작업은 브랜치에 커밋된 상태로 유지
- 재개: 같은 브랜치에서 `/svn-dev`를 다시 실행하면 미완료 단계부터 재개
  - .svn-flow-origin 존재 → 이미 브랜치 생성됨 → Phase 3부터 시작
  - 분석 문서 존재 → Phase 1 생략 가능

## 주의사항

- 각 Phase 시작 시 현재 상태를 출력하여 사용자가 진행 상황 파악 가능
- SVN 커밋은 즉시 서버 반영 — 중간 커밋도 포함
- 분석 문서는 옵시디언에 저장되므로 나중에 참조 가능
- **커밋 메시지에 AI/Claude/자동생성 관련 문구를 절대 포함하지 않음**
